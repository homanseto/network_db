Network DB – Project Structure & Architecture

This document describes the Indoor GIS / Network DB project: structure, system architecture, database schema, workflow, and recommendations. For full detail (every section and table), see PROJECT_STRUCTURE.md.

1. Project Overview

Purpose: Ingest 3D indoor network shapefiles, validate them, enrich with IMDF data (units, levels, openings, building info), and persist into PostgreSQL/PostGIS. Venue is in Postgres; unit, level, opening, and related IMDF layers are in MongoDB (to migrate to Postgres later).

Stack: FastAPI (Python) in Docker; PostgreSQL + PostGIS (EPSG:2326); MongoDB (IndoorMap) for IMDF.

Run: docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build

2. Python over Node.js

Why Python (Shapely) fits this stack

  • CRS / projection (critical for EPSG:2326): Shapely is projection-agnostic and defers CRS to PyProj/Fiona/GDAL. You can keep data in 2326 (meters) and use ogr2ogr or Fiona for ETL with no unnecessary reprojection. Turf is tied to WGS84 (EPSG:4326); every operation assumes degrees. Using 2326 would require constant turf.toMercator / turf.toWgs84, adding floating-point error and overhead—a real risk for precision-sensitive GIS work.
  • Performance and batch processing: Shapely 2.0+ (GEOS via ufuncs) supports vectorized operations and STRtree for spatial indexing. Large batches (e.g. 100k points or spatial joins) scale well. Turf is synchronous JavaScript with no built-in spatial index (no R-tree); it can become slow with very large or complex geometries.
  • Mixed geometry types: The workflow uses points, lines, and polygons together. Shapely/GEOS handles heterogeneous geometry and predicates (intersection, contains, etc.) consistently. Turf has limitations: e.g. some predicates only for same-dimension geometries, and edge cases can be brittle.
  • ETL and ogr2ogr: Python fits the GDAL ecosystem: Fiona reads/writes shapefiles and GeoPackage; GeoPandas can push DataFrames to PostGIS via to_postgis(). You can also call ogr2ogr via subprocess with data already in 2326. In Node, there are no native GDAL bindings; you typically write GeoJSON to disk and shell out to ogr2ogr, which is fragile and slower.

Pros and cons summary

Python (Shapely):

  • Pros: Native 2326 and GDAL workflow; STRtree and vectorized ops; robust mixed geometry handling; clean path from Fiona/GeoPandas to PostGIS; fits FastAPI + ogr2ogr pipeline.
  • Cons: Different language if the rest of your stack is Node; buffer operations may drop Z (check docs if you need 3D).

Node.js (Turf):

  • Pros: Single language if the API is Node/Express; modular packages (e.g. @turf/intersect).
  • Cons: No native projection model for 2326; no spatial indexing (O(n²) risk); GeoJSON-centric and inefficient for large PostGIS loads; ogr2ogr only via temp files and CLI.

Architecture choice in this project

ETL: ogr2ogr (or Fiona) reads/writes shapefiles/GeoPackage in EPSG:2326. Geometry engine: Shapely 2.0+ (vectorized creation, STRtree for spatial joins). API: FastAPI (Python) exposing GeoJSON/endpoints. PostGIS load: GeoPandas to_postgis() or ogr2ogr. This keeps projection, performance, and GDAL in one coherent stack.

Using history (examples)

History tables support point-in-time queries (“time travel”), auditing by ID, and recovery of deleted rows. Example patterns (see SQL/new_db/quey_example.sql for full context):

1. Time travel: restore state as at a specific date

Return the dataset as it was at a snapshot time (e.g. 2026-02-05): current rows that were not changed after that time, plus rows that were later updated or deleted but had a version valid at that time. 

WITH target_date AS (
    SELECT '2026-02-05 23:59:59+08'::timestamptz AS snapshot_time
)
SELECT pedrouteid, shape, feattype, floorid, updated_at
FROM pedestrian_network
WHERE updated_at <= (SELECT snapshot_time FROM target_date)

UNION ALL

-- Restore deletions and pre-update versions from history
SELECT pedrouteid, shape, feattype, floorid, updated_at
FROM pedestrian_network_history h
WHERE h.history_recorded_at > (SELECT snapshot_time FROM target_date)
  AND h.updated_at <= (SELECT snapshot_time FROM target_date)
  AND h.history_id IN (
    SELECT history_id FROM (
      SELECT history_id,
        ROW_NUMBER() OVER (PARTITION BY pedrouteid ORDER BY history_recorded_at ASC) AS rn
      FROM pedestrian_network_history
      WHERE history_recorded_at > (SELECT snapshot_time FROM target_date)
    ) sub
    WHERE rn = 1
  );

2. Audit by business key (e.g. inetworkid) or surrogate key (pedrouteid)

List all changes for one feature: SELECT history_recorded_at, operation, modifiedby, shape, ... FROM indoor_network_history WHERE inetworkid = 'Walkway_L1_001' ORDER BY history_recorded_at DESC;

3. Find recently deleted features

SELECT * FROM pedestrian_network_history WHERE operation = 'DELETE' AND history_recorded_at > NOW() - INTERVAL '7 days';

History design: pros and cons

Pros Cons
Application-agnostic: Trigger runs for any client (API, ArcGIS Pro, QGIS, OpenLayers, Cesium); no need to implement history in each app. Storage: History grows with every UPDATE/DELETE; no automatic retention or purging.
Audit trail: Every change is logged with operation and timestamp; supports compliance and debugging. INSERTs not versioned: The first version of a row is only in the main table until it is updated or deleted.
Point-in-time restore: Queries can reconstruct the dataset as at a past time (see “Time travel” above). Query complexity: Time-travel and “latest version per ID” require careful filters and ROW_NUMBER (or equivalent).
Recovery: Deleted or overwritten rows can be recovered from history. No built-in version labels: No “version 1, 2, 3” or named snapshots unless you add them (e.g. application-level tags).

Import and versioning

How import interacts with history: The import pipeline writes to the main table only (indoor_network or pedestrian_network). It performs INSERT for new rows and UPDATE (upsert) for existing rows. The trigger fires on UPDATE and DELETE, so:

  • New rows (INSERT): No history row is created; the row appears only in the main table.
  • Existing rows (UPDATE via upsert): The trigger runs and inserts the previous version into the history table before the new version is written. So each re-import that changes a row creates one history record.
  • Deletes in the main table (e.g. “remove rows not in staging”) fire the DELETE trigger and log the removed row to history.

Does this fulfill versioning requirements? If users need versioning of the database (audit of changes and ability to query or restore past states), this design fulfills those requirements: every UPDATE and DELETE is captured, and you can run time-travel queries and recover deleted or overwritten data. If they need full snapshot versioning (e.g. “version 2.0 of the whole dataset” with a single label) or scheduled snapshots regardless of changes, you would add application or ETL logic (e.g. periodic dumps or version tags); the trigger-based history is change-based, not snapshot-based.

User edits (ArcGIS Pro, QGIS, OpenLayers, Cesium): When users edit indoor_network directly from these tools, the same database trigger (log_indoor_network_changes()) runs on UPDATE/DELETE and appends the old row to indoor_network_history. No separate application logic is required; the DB captures all edits regardless of client.

Workflow feedback: This flow aligns with the project: staging → validate → enrich → main table keeps the main table clean and allows validation to block bad data. One improvement is to document in runbooks that ArcGIS Pro, QGIS, and OpenLayers connect directly to Postgres so operators know the same trigger applies to all edits. Idempotent re-runs (same ZIP/path) are already supported by truncating staging and upserting by inetworkid.

6.2 Export Workflow

Export reads from indoor_network by venue displayname and writes Shapefile and/or GeoJSON (via ogr2ogr). Two API entry points are used:

  1. GET /export-indoor-network/ — Exports to a path (or default result dir). Parameters: displayname, export_type (indoor, pedestrian, or all), export_format (shapefile or geojson), output_dir (optional). The service builds a SQL SELECT using pedestrian_convert_table.json for field mapping, filters by displayname (and restricted = 'N' when opendata=open), then runs ogr2ogr: Shapefile stays in EPSG:2326; GeoJSON is transformed to EPSG:4326 and coordinates are rounded to 8 decimals. Returns JSON with status and output path.
  2. GET /download-indoor-network-zip/ — Returns a ZIP containing both formats. Parameters: displayname (required), type (pedestrian, indoor, or all), opendata (open or full). The API exports to temp dirs (SHP and GeoJSON), zips them, and streams the ZIP as a download; temp dirs are removed afterward.

Flow summary: Request (displayname + options) → service export_indoor_network_by_displayname() → load field mapping → SQL on indoor_network → ogr2ogr → Shapefile and/or GeoJSON (and optionally zip + stream). No staging table; export is read-only from the main table.

6.3 API interface: Swagger

The API interface for import and export is described in api/app/reference/swagger.yaml. Users can use this OpenAPI definition (or the served Swagger UI) to discover and call the endpoints.

  • Import: POST /import-network-upload/ — multipart form with files (ZIPs). Display name is derived from each ZIP filename; each file should contain 3D Indoor Network.shp. Responses: 200 with import result list, 500 on failure.
  • Export: GET /download-indoor-network-zip/ — query parameters displayname (required), type (pedestrian | indoor | all), opendata (open | full). Returns a ZIP containing a Shapefile folder and a GeoJSON folder for the given venue. 200 returns the binary ZIP; 500 on export error.

The same swagger.yaml also documents other Indoor API paths (e.g. checkSubmission, duplicateCheck, download/groupBuildingInfo, updateUnitOrPOIInfo, fullversion, opendata). For network import and export, the two paths above are the ones used by users.

6.4 Staging → main table → history (network and pedestrian)

Data is never imported directly into history tables. History is written automatically by database triggers when rows in the main table are updated or deleted.

Network data flow

  1. Staging: The shapefile is loaded into network_staging by ogr2ogr (one-time load per import). Columns (e.g. INETWORKID, shape, highway, oneway, flpolyid) come from the shapefile. The table is truncated before each new run.
  2. Validation: The PL/pgSQL function validate_network_staging() runs. It checks geometry NOT NULL, ST_IsValid, SRID = 2326, ST_NDims = 3, INETWORKID NOT NULL, and no duplicate INETWORKID. Failures go to network_staging_errors. If invalid, the API returns errors and does not move data to the main table.
  3. Main table: After validation passes, the application reads rows from network_staging, validates with Pydantic (NetworkStagingRow), enriches them (feattype, wc_access, alias names, level_id, floorid, etc.), then upserts into indoor_network. Staging is then truncated.
  4. History: indoor_network_history is populated only by a trigger on indoor_network: before UPDATE or DELETE, log_indoor_network_changes() inserts the old row into indoor_network_history. There is no separate “import into history”.

Pedestrian data flow

  1. Staging: Pedestrian data is loaded into pedestrian_staging from FGDB via ogr2ogr (layer PedestrianRoute, -t_srs EPSG:2326).
  2. Main table: merge_staging_to_production(staging_table) maps staging columns to pedestrian_network via pedestrian_convert_table.json, then UPSERT (ON CONFLICT pedrouteid DO UPDATE) and DELETE rows not in staging. The main table is pedestrian_network.
  3. History: pedestrian_network_history is filled only by a trigger on pedestrian_network: log_pedestrian_network_changes() inserts the old row before UPDATE/DELETE.

Summary: Staging is the landing zone for raw import; the main table (indoor_network or pedestrian_network) is the target of processed data; history tables are append-only audit logs maintained by triggers on the main table.

Why use triggers for history?

Users edit indoor_network from multiple tools: QGIS, OpenLayers, Cesium, and other custom applications. Because many users may edit the same data concurrently, the system cannot rely on a single application to write history. The database must automatically record changes whenever rows are updated or deleted, regardless of which client (ArcGIS Pro, QGIS, API, or another app) performed the edit. Triggers on indoor_network (and pedestrian_network) ensure that every change is logged to the corresponding history table without requiring each application to implement history logic.

This network-db import API is one part of the system. You also have other applications for editing units, openings, and points. You can refer to each deployable, bounded-capability component (this API, the QGIS workflow, the OpenLayers/Cesium editors, etc.) as a service or microservice if that fits your architecture vocabulary. The important point: the history trigger lives in the database, so it is application-agnostic and correctly captures edits from any client.

6.5 Logic for calculating each network (geometry and derived fields)

How geometry is used and how each derived field is computed for a network row (indoor network import).

Source of geometry

The shape comes from the shapefile. ogr2ogr loads it into network_staging as PostGIS LineStringZ, SRID 2326. The application reads with ST_AsGeoJSON(shape) AS geojson (coordinates in 2326). The geometry is read for spatial calculations and written to indoor_network (via WKT from the same GeoJSON).

Derived fields

Field Logic
level_id Level features (MongoDB): match row.flpolyid to level.properties.FloorPolyID → that level’s id.
floorid flpolyid_slices(flpolyid) → buildingCSUID, floorNumber. BuildingInfo (Mongo) by buildingCSUID → SixDigitID. floorid = SixDigitID + floorNumber.
bldgid_1, buildingnameeng, buildingnamechi From same BuildingInfo: BuildingID, Name_EN, Name_CH.
levelenglishname, levelchinesename From level feature (matched by flpolyid): properties.name.en, properties.name.zh.
feattype calculate_feature_type(row, unit_features, unit3d_features): Line from geojson → transform to 4326; filter units by level_id; find unit polygon with longest intersection length with line; map unit category (escalator, ramp, stairs, etc.) to feattype code (1=walkway, 8=escalator, 10=lift, 11=ramp, 12=stairs, 13=stairlift). Geometry: line–polygon intersection in 4326.
gradient calculate_gradient(geojson): vertical change (delta Z) and horizontal length from 3D line → angle (e.g. atan2).
wc_access calculate_wheelchair_access(row, opening_features): If feattype ≠ 11 (ramp) return 2. Else: line to 4326; filter openings by level_id; buffer each opening by 0.1 m; if line intersects any buffer → 1, else 2.
wc_barrier 1 if feattype in (8, 12) or wheelchair == 'no'; else 2. No geometry.
direction From oneway: 'no' → 0, 'reverse' → -1, 'yes' → 1.
emergency 'no' if feattype == 10 (lift), else 'yes'.
aliasnamen, aliasnamtc get_alias_name(row, opening_features): line to 4326; openings with name, same level; if line intersects 0.1 m buffer of an opening → use opening name + building + facility name; else building + level/facility. Chinese: spaces removed. Geometry: line–buffered-opening intersection.
shape_len Trigger update_shape_len() on indoor_network sets shape_len = ST_3DLength(shape) on INSERT/UPDATE. Computed in PostGIS.

In short: Geometry (LineStringZ, 2326) is read from staging, transformed to 4326 for comparisons with MongoDB units/openings, and used to compute feattype (intersection with units), wc_access and alias names (intersection with buffered openings), and gradient (Z and length). level_id, floorid, and building/level names come from flpolyid and lookups, not from geometry.

7. Pros and Cons

Pros: Clear separation (routes → services → schema); single import pipeline for ZIP/path; strong validation (Postgres + Pydantic); audit (history tables); venue in Postgres with FK; reference docs (pedestrian_convert_table.json, SQL); Docker one-command run.

Cons: Dual store (MongoDB still required for unit/level/opening/BuildingInfo); blocking I/O in async path; hardcoded /import-network/ route; network_staging schema from ogr2ogr not DDL in repo; mixed error response shapes.
